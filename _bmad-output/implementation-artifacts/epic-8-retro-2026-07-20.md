## Successes

1. **100% story completion on the highest-risk epic attempted.** 4 stories in 4 days, zero production incidents. Upgraded the entire Go + Kubernetes dependency stack spanning 7 minor versions of controller-runtime.

2. **Largest single refactor in project history executed systematically.** 44 webhook files migrated from `webhook.Defaulter`/`webhook.Validator` to `webhook.CustomDefaulter[T]`/`webhook.CustomValidator[T]` using a "start with simplest type, nail the pattern, apply everywhere" strategy. Zero compilation errors at the end.

3. **Phase 1 test stabilization investment paid off.** The test suite (unit + integration) was the safety net for this upgrade — envtest upgraded from K8s 1.29 to 1.36, Kind from v0.27 to v0.32, and all existing tests passed without needing test code changes.

4. **Just-in-time version audit practice proved its value for the 2nd consecutive epic.** During the D4 retro, the team caught that Epic 8's original targets (Go 1.24, CR v0.23, K8s v0.35) were stale — Go 1.24 had reached EOL. Retargeted to Go 1.26 / v0.36 generation before any work began.

5. **Story intelligence chain at peak performance.** Each story's dev notes explicitly documented what was deferred to the next story (Scope Boundary + Previous Story Intelligence sections). Story 8.4's notes referenced exactly what 8.1, 8.2, and 8.3 had each contributed and deferred. No context was lost.

6. **Code review continued as primary quality gate.** 8 findings across all 4 stories — CRD schema issues, RBAC migration gaps, stale binaries, documentation inconsistencies. All caught before merge. 5th consecutive epic with meaningful review findings.

7. **Pre-existing issues properly separated from story scope.** In Story 8.2's review, 2 of 4 findings were tagged as pre-existing and deferred — not mixed with the story's own changes. This discipline keeps stories focused and prevents scope creep.

## Challenges

1. **Chicken-and-egg dependency between Stories 8.2 and 8.3.** Story 8.2 couldn't regenerate CRDs because controller-gen v0.14.0 was incompatible with k8s.io/api v0.36.0 — but the controller-gen upgrade was Story 8.3. Had to revert CRDs to HEAD in 8.2 and defer regeneration to 8.3. Managed well via explicit documentation but added complexity.

2. **GOTOOLCHAIN=local friction in Story 8.1.** The system Go refused to download the toolchain automatically. Resolved by switching to `GOTOOLCHAIN=auto`, but consumed debugging time.

3. **Kind cluster flakiness continues.** The Kind cluster container was stopped (USB device stale mapping issues) during Story 8.1, requiring cluster recreation. This is a recurring pre-existing issue, not Epic 8-specific.

4. **Mount path composition documentation missed from D4 retro.** 1 of 2 D4 action items not followed through — the mount path convention was not documented in project-context.md. No natural trigger existed in a dependency upgrade epic. Remediated in this retro.

---

## Key Insights

1. **Sequential dependency chains need explicit "deferred to Story X" documentation.** The 8.1→8.2→8.3→8.4 chain had hard dependencies. Each story's Scope Boundary section listed what was NOT in scope and which future story owned it. This eliminated ambiguity and prevented duplicate or missed work.

2. **Systematic migration strategies are essential for large-scope refactors.** The 44-webhook migration succeeded because of the pattern: migrate the simplest file first (Policy), validate the pattern compiles and works, then apply it across all remaining files. This approach is reusable for future large refactors.

3. **Pre-existing issues should be tagged and deferred, not mixed with story scope.** This practice was confirmed across multiple stories — it prevents scope creep and keeps reviews focused on the story's actual changes.

4. **Small debt items should be remediated immediately during retros.** The mount path documentation slipped from D4 retro through all of Epic 8. This retro remediated it plus 2 other small items (webhook markers, loggers) immediately rather than carrying them further.

5. **Just-in-time version audits are now a 2x-validated practice.** Both Epic 8 and Epic 9 had stale targets when audited. For Epic 9, the audit discovered Vault 1.21 EOL and triggered a retarget to Vault 2.0.3 — a major scope adjustment caught before any work began.
