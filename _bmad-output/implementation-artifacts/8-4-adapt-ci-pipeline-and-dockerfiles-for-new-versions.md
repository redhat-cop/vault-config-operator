# Story 8.4: Adapt CI pipeline and Dockerfiles for new versions

Status: ready-for-dev

## Story

As an operator developer,
I want the CI workflows and Dockerfiles to reflect the upgraded Go, controller-runtime, K8s, and tooling versions,
So that builds, tests, and releases work correctly with the updated stack from Stories 8.1–8.3.

**What the reusable workflow does with these inputs:**
1. **Setup job:** Installs Go (`actions/setup-go` with `GO_VERSION`), downloads operator-sdk binary at `OPERATOR_SDK_VERSION`, runs `make controller-gen` to install controller-gen at the Makefile's `CONTROLLER_TOOLS_VERSION`
2. **Build job:** Compiles the operator via `make` for each platform, builds Docker image via `ci.Dockerfile` (pre-built binary copy — no Go needed)
3. **Bundle job:** Runs `make bundle`, validates bundle via `operator-sdk bundle validate`
4. **Test job:** Runs `make test` (unit tests) and/or `make integration` (integration tests)
5. **Helm job:** Runs `make helmchart-test`

The CI does NOT use `Dockerfile` — it uses `ci.Dockerfile` (copies pre-built binary). The `Dockerfile` is for local development builds only.

### OPERATOR_SDK_VERSION Compatibility Risk

**Key risk:** operator-sdk v1.31.0's `bundle validate` command may not support CRD schemas produced by controller-gen v0.21.0. controller-gen v0.21.0 may emit markers or schema properties (e.g., `k8s:enum`, `k8s:immutable`, CEL validation rules) that v1.31.0 doesn't recognize.

**Mitigation:** Run `make bundle` locally first. The `bundle validate` step will either pass or fail clearly. If it fails:
1. Check the specific validation error — it's usually about unrecognized CRD schema fields
2. Try `operator-sdk bundle validate ./bundle --select-optional name=operatorhub` (the CI reusable workflow uses this more permissive validation)
3. If basic validation fails, the minimum SDK version that supports CRDs from controller-gen v0.21.0 needs to be identified. Operator SDK v1.37+ should support k8s.io/* v0.35+ CRD schemas
4. If upgrade is needed, keep it minimal — find the lowest version that passes validation rather than jumping straight to v1.42

**Most likely outcome:** operator-sdk v1.31.0 bundle validation is lenient about extra CRD schema properties — it validates the bundle structure (manifests, metadata, annotations) rather than deeply inspecting CRD OpenAPI schemas. This should pass without issues. However, the `operator-sdk generate kustomize manifests` step may produce different output with different SDK versions, which is why the full SDK upgrade is deferred to Story 10.1.

### CI Pipeline End-to-End Flow

After all Epic 8 changes, the CI pipeline should work as follows:

1. **Setup:** Install Go 1.26, download operator-sdk v1.31.0, install controller-gen v0.21.0 (from Makefile)
2. **Build:** Compile with Go 1.26, build multi-arch images via `ci.Dockerfile`
3. **Test:** `make test` uses envtest K8s 1.36.0 binaries; `make integration` uses Kind v0.32.0 with K8s 1.36.1 nodes
4. **Bundle:** `make bundle` generates with controller-gen v0.21.0 and validates with operator-sdk v1.31.0
5. **Helm:** `make helmchart-test` creates Kind cluster, deploys, and tests
