## Successes

1. **Excellent velocity through pattern reuse** — Story 1.1 established the testing approach (standard Go `testing` package, table-driven subtests, document-behavior-as-is for AC#4). Stories 1.2–1.6 scaled this pattern to all 46 CRD types in roughly two days.

2. **Effective code review process** — 23 findings across 6 stories, all either patched within the story or properly deferred to the correct future story. Reviewers consistently caught real quality issues (self-referential payloads, missing edge cases), not nitpicks.

3. **Massive coverage improvement** — `api/v1alpha1` coverage went from 3.0% to 16.9%, a 5.6x improvement providing a solid safety net for future changes.

4. **Proactive debt resolution** — Raffa manually fixed 2 of 3 technical debt items (GroupAlias `fmt.Print` debug statements, KubernetesSecretEngineRole field-to-key mapping swap) before the retrospective.

5. **Early discovery of AC#4 contradiction** — Story 1.1 identified that ~38 types use bare `reflect.DeepEqual` causing unnecessary writes when Vault returns extra fields. This was documented immediately and tracked in Story 7-4 rather than being silently ignored.

---

## Challenges

1. **Self-referential test payloads** — In stories 1.2 and 1.3, the dev agent built expected Vault payloads by calling `toMap()` on the same object being tested, proving only `x == x`. Caught by code review both times but indicates a systemic gap in guidance.

2. **Missing negative test cases for custom behavior** — Stories 1.2 and 1.5 were missing explicit negative tests proving custom `IsEquivalentToDesiredState` logic (e.g., full mount spec rejected for tune-only types, payloads containing redacted keys). These are the highest-value tests and were absent from initial implementation.

3. **Story spec vs actual code discrepancies** — Story 1.4 specified incorrect key counts (27 for JWTOIDCRole, actual 26; 22 for GCPRole, actual 21). The dev agent rewrote task text to match implementation instead of documenting the discrepancy.

4. **Unrelated formatting changes in diffs** — Stories 1.4 and 1.6 had `go fmt` reformatting files from previous stories leaking into their diffs, making review harder.

---

## Key Insights

1. **Pattern-first investment pays dividends** — Spending time in Story 1.1 to nail the testing approach paid off massively in subsequent stories.

2. **Code review is the quality backstop** — Every major issue was caught in review, not in production. The review process works.

3. **Lessons not captured in `project-context.md` will repeat** — The self-referential payload and missing negative case issues recurred across multiple stories because no permanent rule existed to prevent them.
