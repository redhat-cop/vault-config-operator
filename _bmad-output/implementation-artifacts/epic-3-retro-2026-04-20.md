## Successes

1. **Cleanest epic yet** — Zero debug failures across all 4 stories. Every implementation was clean on the first pass. This is a direct result of mature story specs and established patterns.

2. **Pattern compounding across three epics** — Epic 1 established unit test patterns, Epic 2 established integration test patterns, Epic 3 scaled them to new types mechanically. Review patches trended down (2→3→2→1), confirming the dev agent was learning from prior stories within the epic.

3. **Both reconciler variants now covered** — VaultResource (Policy, PasswordPolicy) and VaultEngineResource (SecretEngineMount, AuthEngineMount). This is important because Epic 4's auth engine types build directly on the VaultEngineResource pattern from Stories 3.3–3.4.

4. **Minimal infrastructure churn** — Only 1 decoder method added (`GetAuthEngineMountInstance` in Story 3.4). All other decoder methods, controller registrations, and Vault auth roles already existed. The Epic 2 readiness assessment was accurate.

5. **Consistent model quality** — All 4 stories used Opus 4.6 exclusively. Combined with the Epic 2 retro recommendation to prefer Opus-class models, this produced uniformly clean implementations.

---

## Challenges

1. **Vault client `json.Number` type handling** — Stories 3.3 and 3.4 required `json.Number` type assertions for tune config TTL values (Vault returns integer seconds, not duration strings). This was a Vault Go client quirk that wasn't anticipated in story specs but was handled cleanly.

2. **Review findings concentrated on type safety** — 3 of 4 stories had unsafe `.(string)` type assertions that would panic on test failure instead of producing a helpful Gomega error. This is a recurring pattern the dev agent should internalize: always use checked type assertions in test code.

3. **No real blockers** — Honestly, Epic 3 was straightforward. The challenge shifts to Epic 4 where external service dependencies (LDAP, OIDC) introduce real infrastructure complexity.

---

## Key Insights

1. **When story specs are detailed and patterns are established, implementation becomes mechanical.** Epic 3 validated that the story spec + code review process is now mature for the "simple types" category. The challenge shifts to *new territory* in Epic 4.

2. **Review patch trend (2→3→2→1) suggests in-epic learning.** The dev agent applied lessons from earlier stories to later ones within the same epic. Story 3.4 had only 1 review finding — the lowest in project history.

3. **VaultEngineResource is fundamentally different from VaultResource.** The tune-only comparison, mount existence check via sys/mounts|sys/auth, accessor population in status, and json.Number TTL handling are all patterns that must carry forward to Epic 4's auth engine config/role types.

### Epic 4 Infrastructure Decision: Keycloak for JWT/OIDC (Story 4.3)

Per Raffa's direction, Story 4.3 will use **Keycloak as a real OIDC provider** instead of a mock JWKS endpoint:

- Deploy Keycloak to Kind using the [Keycloak Operator](https://www.keycloak.org/operator/installation) (kubectl-based installation without OLM)
- Configure a Keycloak realm and client for Vault JWT/OIDC auth
- This follows the project's "install in Kind" philosophy and tests the actual JWT/OIDC flow end-to-end
- Story 4.3 spec must explicitly flag Keycloak deployment as infrastructure scope (similar to PostgreSQL in Story 2.4)
