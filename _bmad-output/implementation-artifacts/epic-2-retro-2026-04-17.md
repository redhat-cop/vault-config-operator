## Successes

1. **Pattern-first investment paid off again** — Story 2.1 established the update test pattern (Get → Update → Poll Vault → Verify ObservedGeneration), and Stories 2.2–2.4 scaled it across 4 fundamentally different reconcile mechanisms (VaultSecret's `shouldSync`, RandomSecret's refresh guard, Entity's standard `VaultResource`, PKI's `VaultPKIEngineResource`) without reinventing the wheel.

2. **Story 2.0 as a force multiplier** — Investing in idempotent infrastructure before any test stories meant zero flakiness from re-runs across the entire epic. Vendored ingress-nginx, namespace create-or-get, configurable ports — all compound benefits.

3. **Real bugs found through end-to-end coverage** — The `hclsimple.Decode` filename bug (Story 2.2), the `vaultAddress` stale reference bug (Story 2.0), and the pre-existing swapped TTL assertions (Story 2.1) were all latent production defects that no prior testing had caught. The `hclsimple.Decode` bug meant inline password policies were entirely non-functional — the feature was dead code in production.

4. **Code review remained a strong quality backstop** — 14 patches across 4 stories, all substantive (type safety, assertion completeness, verification correctness). Zero nitpicks. Every major issue was caught in review, not at runtime.

5. **Full follow-through on Epic 1 action items** — 4 for 4 on commitments from the previous retrospective.

## Challenges

1. **Story 2.4 scope underestimation** — DatabaseSecretEngineStaticRole required new infrastructure (PostgreSQL deployed to Kind via Bitnami Helm chart) that wasn't anticipated in the epic definition. The existing integration test file set up all database engine infrastructure but never actually created a static role. The story was really "build infrastructure + create test + update test," not just "add update scenario."

2. **Model quality variance** — Story 2.3 (Sonnet 4.5) had the highest review finding count (5 patches, 7 deferred) compared to Opus-implemented stories. Bare type assertions that would panic, incomplete assertion coverage, and defensive coding gaps were the main delta.

3. **CreateOrUpdateConfig dual bug in PKI engine** — Discovered during Story 2.4 analysis. Two bugs in `vautlpkiengineobject.go:105-118`: (a) write path hardcoded to `GetConfigCrlPath()` instead of using the `configPath` parameter — URL config writes go to the CRL path; (b) `IsEquivalentToDesiredState` compares URL/CRL config payload against `PKICommon.toMap()` (CA generation payload) — completely different field sets, always returns false, causing unconditional writes every reconcile. Documented and deferred.

4. **Type system surprises in story specs** — Story 2.4's `MaxTTL` was specified as `string` ("4380h") in the tasks but the actual Go type is `metav1.Duration`, causing a build error during implementation.

---

## Key Insights

1. **Update tests are the highest-value integration tests** — They exercise code paths that create/delete tests never touch, and they found 3 real bugs in 5 stories. The `hclsimple.Decode` bug was a complete feature breakage that had shipped to production undetected.

2. **Infrastructure scope must be called out explicitly in planning** — When a "test story" requires deploying a new service to Kind, that's infrastructure work that should be flagged in the story definition, not discovered during implementation.

3. **Model selection matters for complex types** — Identity system types with nested `interface{}` maps and Vault-specific response shapes benefit from stronger models. This should inform story assignment.
