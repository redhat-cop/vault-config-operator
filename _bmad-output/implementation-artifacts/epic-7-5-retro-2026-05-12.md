## Successes

1. **Phase 1 safety net ROI proven.** This was the first epic touching business-facing CRD schemas (as opposed to test-only or hardening changes). 170+ field annotation changes across 31 files shipped with zero regressions — directly enabled by the 90 integration specs and comprehensive unit tests built in Epics 1-7.

2. **Enormous scope delivered in 3 days.** 8 stories, 170+ field changes, 31+ type files, 30+ validation markers added, 30+ YAML test fixtures updated, 24 integration test files refactored. 100% completion.

3. **Story 7.5.3a — systemic infrastructure fix born from pattern recognition.** The team discovered that removing `omitempty` (R2) causes typed Go clients to bypass CRD server-side defaulting. Instead of patching each story, they extracted the `CreateFromYAML` unstructured fixture method as a permanent infrastructure improvement — then reverted all workarounds from Stories 7.5.1-7.5.3.

4. **Dramatic learning curve.** Stories 7.5.1-7.5.3 accumulated 5/7 debug failures and 3/6 review findings. Stories 7.5.4-7.5.7 had just 2 debug failures and 3 findings, with 7.5.4 and 7.5.5 achieving zero-defect execution.

5. **Review finding quality matured.** 0/6 findings were assertion weakness (vs 10/15 in Epic 7). All 6 were architectural decisions — typed-client contract change (7.5.3), project-context update (7.5.3a), RandomSecret V1 defaulting (7.5.6).

6. **Story 7.5.7 added 30 admission-time validation markers.** 8 Enum + 22 Minimum/Maximum markers now catch invalid values at Kubernetes admission before they reach the operator or Vault API.

7. **Final audit confirmed zero R1/R2 violations.** Story 7.5.6 Task 8 grep confirmed zero remaining annotation rule violations across the entire `api/v1alpha1/` codebase.

## Challenges

1. **R2 omitempty + CRD defaulting interaction — unpredicted.** Removing `omitempty` from JSON tags caused typed Go serialization to send empty strings for absent YAML fields, bypassing CRD server-side defaulting and triggering Enum validation failures. Hit Stories 7.5.1, 7.5.2, and 7.5.3 before Story 7.5.3a fixed it systemically. 5 of 7 debug failures traced to this root cause.

2. **Kind cluster infrastructure degradation.** Integration test runs sometimes left namespaces stuck in Terminating state, requiring fresh cluster recreation. Affected Stories 7.5.3 and 7.5.6.

3. **Cross-cutting repetition amplified by structural duplication.** The same token tuning fields (`TokenType`, `TokenNumUses`, `TokenPeriod`, etc.) appear in 6+ role types with nearly identical definitions. Applying the same annotation changes across all of them was tedious and error-prone — a direct consequence of the codebase's structural duplication.

---

## Key Insights

1. **Cross-cutting refactors expose structural duplication.** Epic 7.5 generated actionable evidence for Epic R1 — when the same `TokenType` Enum is added to 7 files, that's a signal those types share structure that should be factored out. R1-4 (test decoder generics) and R1-5 (reconciler deduplication) address exactly this.

2. **Test safety nets pay their first dividend when you touch business-facing code.** Epics 1-7 built the safety net. Epic 7.5 is where it proved its value — catching R2/defaulting regressions that would have silently shipped as behavioral changes to users.

3. **Emergent stories are healthy adaptation.** Story 7.5.3a was unplanned, but detecting and fixing a systemic infrastructure problem mid-epic is a strength, not a planning failure.

4. **Annotation-only refactors are NOT risk-free.** Despite zero Go logic changes, the CRD OpenAPI schema shifts had real implications for typed client serialization, server-side defaulting, and admission validation. The test safety net was essential.
